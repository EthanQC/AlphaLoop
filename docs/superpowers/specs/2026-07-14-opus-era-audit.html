存量代码专项审计报告（2026-07-14）

背景：用户要求对 v2 重建之前的存量脚本（早期由 Opus 产出）做格外检查。

方法：49-agent 对抗式审计 workflow——33 个存量脚本按 7 条链路分组，每组一个带定制视角的审计 agent，每条发现再派一个独立反驳者尝试推翻（读真实代码、可对临时库跑验证），驳不倒才算数。已修复/已排期的已知问题预先排除，不重复上报。

结果：确认 38 项（Important 21 / Minor 17），驳回 4 项。全量发现（含完整触发场景与验证推理）：.superpowers/sdd/audit-2026-07-14-confirmed.json。

一句话总评

存量代码的主功能路径基本正确，但普遍缺"失败路径工程"：错误被静默吞掉、降级不披露、状态写坏无察觉——与 Phase 2.5 在提醒器上连修五轮的问题同属一类。最严重的集中在下单-对账链路（P6 要在上面建审批）和"绿色假象"类（系统看起来正常、实际在输出坏数据）。

重点发现（按链路）

1. 下单-对账链路（money path）——4 项 Important，全部记为 P6 前置

这是含金量最高的一组，不做零散修补（改一处会让另一处更糟），P6 建审批链路时统一重设计：

发现要害
提交路径无任何幂等下单成功但记录前崩溃/超时 → 用户重试 → 同一笔单在券商重复下出；执行器先执行后落库，先天顺序反了
对账查询的报告标题从来没有代码写过Execution report for X vs 实际写入的 X 执行报告——一次改名（f2984a5）漏改查询 → 订单和 ticket 的关联从未建立过，全部 Ticket: unknown
关联逻辑是"同 symbol 最新者胜"修好上面的标题反而更糟：卖单 ticket 会经 COALESCE 覆盖买单已正确的关联，审批/审计挂错单
10% 预算守卫只看已成交持仓未成交限价单不可见 → 预算可以被承诺两次

Minor 三项（币种混算、撤单状态映射成 unknown、对账每跑一次重复插一条 trade 报告虚增交易数）同记 P6。

给 P6 的结论：submit→record→reconcile 链条必须以幂等键 + 先记录后执行重建，不能在现骨架上贴补丁。

3. 调度与状态存储（并入 Task 5，与 H1 同一教训）

  • processed-runs.json 非原子写，损坏即静默清空全部去重与停机标记——历史任务全量重放、已停机任务自动复活；
  • 状态文件写不进去（磁盘满/权限）无升级路径——任务执行了却记不上，KeepAlive 下重启即重跑重发；
  • 失败分类拿含时间戳的 stderr 首行当 key——同根因永远 count=1，3-strikes 停机永不触发（与 Task 5 已排期项撞车，互相印证）；
  • 超时只发一次 SIGTERM 无 SIGKILL 兜底；
  • 两个安装器互相打架：launchd:install-user 会重装另一个安装器刚 retire 的 5 个任务，报告双份执行。

4. 报告管线（Important 并入 Task 7；注入类记 P4）

  • 投递完整性标记「长桥行情」从不出现在报告里 → 每次投递都判不完整、强制重新生成整份报告（写侧/读侧各测各的，又是 seam 类）；
  • 宏观日历抓取失败无降级 → 整份日报直接失败（其他源都有降级）；
  • 记给 P4（重建新闻引擎时一并）：RSS 实体解码顺序会把合法转义还原成活 HTML、无日期旧闻被标"刚刚发布"排榜首、质量门的来源多样性可被标题注入伪造、新闻标题的 markdown 链接语法直通 PDF。

5. 凭据与飞书链路（并入 Task 7）

  • oauth 成功一次就写坏 .env.local（值不加引号/剥引号/毁注释）→ 直接让 H2 刚接线的安装器中断——两个写回点同病；
  • 插件用 npx -y 不固定版本冷启动——registry 给什么跑什么，进程环境里是全部飞书密钥（供应链面）；
  • Minor：oauth 无 state 参数（P10 加固）、id 缺失的飞书事件全部塌缩到主键 "" 互相覆盖。

6. 个股分析 CLI（并入 Task 7）

  • 无逐 symbol 隔离——一个标的行情失败杀掉整批分析且无限期；
  • 显式降级态（仅长桥新闻）被自己的质量门恒拒——整批新闻降级时永远发不出报告（死锁式降级）；
  • 「180 日均线」用 6 个月数据算（最多 ~126 根）——标签与事实不符；
  • Minor：期权到期日当天跳到下月、带点的分级股票代码不转 Yahoo 横线格式（5 个数据源全 404）、prepare 静默覆盖已投递的报告归档。

被驳回的 4 项（反驳者证伪，不采纳）

嵌入 JSON 解析取首段（实际实现取的是最后一段有效 JSON）；对账重跑会改写历史关联（COALESCE 方向判断有误，历史非空值不会被覆盖——但"首次即错"的版本成立，见上）；5 秒轮询无防重入（inFlightRunKeys 实际按积压 run 逐条去重有效）；wrapper 复活旧 OAuth token 遮蔽轮换后的 refresh token（实际插件自持久化优先）。
