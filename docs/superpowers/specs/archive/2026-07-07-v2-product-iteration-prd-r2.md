## 1. 现状梳理（2026-07-07 实测确认）

### 1.1 系统今天是什么

仓库里已经有一套完整、带测试、护栏齐全的"四能力"代码：

| 能力 | 今天的形态 |
|---|---|
| 中文日报/周报 | 每天/每周定时生成，Markdown→PDF，发到飞书群（摘要+PDF 附件） |
| 个股批量分析 | 每 3 天对标的池逐股生成分析报告（8 段固定结构），同样 PDF 进群 |
| 模拟盘监控与下单 | 长桥官方模拟盘：每小时快照、收支报告；下单必须经本机 broker-executor，服务端强制"仅模拟盘、仅股票/ETF、仓位 ≤10%、实盘/期权永远拒绝" |
| 飞书群 @ 回复 | 群里 @ 机器人可以用中文对话、让它跑分析、查持仓 |

这些代码的质量经这次评审逐行核实：**执行护栏全部真实有效**（不是口头承诺，是服务端代码强制），报告管线、质量门、重试机制都在。这是可以放心承重的地基。

### 1.2 但现在什么都没有在跑

机器用户迁移（mashu→abble）之后，整套运行时从未重装：

- 三个守护进程（网关、定时任务执行器、下单服务）全部没有安装，最后一批报告停留在 2026-06-19；
- 环境变量文件 `.env.local` **在迁移中丢失了**——长桥、飞书的全部凭据需要从旧机或密码库找回重配，这可能是重新点火中最花时间的一步；
- 数据库文件也不存在了，首次点火后需要跑约 90 分钟才能积累出下单所需的新鲜账户快照。

所以 R0（点火）不是"检查一遍"，而是一次**从零重装 + 凭据重建 + 重启演练**。

### 1.3 评审补进需求的三件事（已并入各 R，无需你决策）

这次评审发现三块"所有功能都依赖、但上一版 PRD 没写"的地基，已作为需求的一部分补进对应章节：

1. **可交互的飞书卡片**：系统至今只发过纯文本和 PDF 文件，从来没发过"带按钮的卡片"。而 v2 的报告卡、提醒卡、审批卡全是按钮卡片——这个能力先建，三处共用。
2. **数据的保险箱**：v2 之后，你的每一次审批、每一条预测、每一次提醒都存进本机数据库，成为月度复盘的唯一原料。上一版没有为它安排备份——现在明确：每天自动备份、保留多天、上线前做一次"从备份恢复"演练。丢了这个库=复盘飞轮全部白积累。
3. **报告站点的"服务员"**：手机点开报告链接，背后需要一个把页面递给浏览器的常驻小服务。上一版只写了"有个站点"，没写这个服务由谁提供——现在作为 R2 的明确交付物。

### 1.4 一条好消息

上一版最大的技术悬念是"审批卡片的按钮点击能不能收到"。评审实际检查了本机安装的 OpenClaw：**原生支持**——按钮点击会自动进入机器人的对话管线，还自带"只认你本人点击、过期作废、防重复点击"。所以 R6 的审批体验（卡片上点批准/拒绝）**照原计划做，不缩水**，只需点火后花几分钟实测确认两处配置。

## 3. 需求变更提议（需要你逐条确认）

> 规则：调研结论（见 2026-07-03 调研报告）全部尊重，砍或调的每一条都在下面讲清楚"最终效果差什么、为什么现在不做、什么时候回来做"。你逐条批 ✅ / 驳 ❌，驳回的按原 PRD 执行。

### 3.1 建议移出本期（2 项）

**C1｜个股报告的"五因子同业字母分级"（原 R5.4）**

- **效果差异**：报告里不会出现 Value/Growth/Profitability/Momentum/Revisions 的 A-F 字母评级和"单因子太差一票否决"结论。仍然会有基于该股自身数据的强弱标签（如"估值偏贵、动量走强"）和结构化结论框。
- **为什么现在不做**：字母分级的定义是"**和同行业公司比**排百分位"。免费数据源只能拿到你正在分析的这只股票的数据，拿不到全行业几百家公司的面板——调研借鉴的 Seeking Alpha 背后是整个市场的付费数据库。没有同业基准硬做出来的 A-F 是"假分级"：看着专业，实际会误导你。调研的价值保留在"因子思维进结论框"，砍的只是"同业相对+字母"这个做不出真货的呈现形式。
- **什么时候回来**：找到可用的同业面板数据源（付费或新的免费源）时即可加回——系统已预留扩展空间，不需要推倒重来。

**C2｜"机构持仓变化"数据（原 R5.1 的 SEC 13F 部分）**

- **效果差异**：个股报告的事实表里没有"机构增减仓"栏目，该位置显式标注"数据不可得（原因：无免费实时源）"。报价、估值、基本面、分析师目标价、空头仓位、财报日历全部照做。
- **为什么现在不做**：13F 是机构按**季度**申报、允许滞后 45 天——你看到的"最新"机构动向实际是 1.5～4.5 个月前的旧闻，对波段决策基本没有参考价值。而且它按"机构"申报而不是按"股票"，要算出"NVDA 被机构增持了多少"得自建一个批量解析数千份季度申报的工程。投入是一个独立项目，产出是滞后数据。
- **什么时候回来**：出现付费聚合源（一次调用直接拿单股数据）、或你明确需要季度级机构动向时。

### 3.2 建议调整规格（2 项）

**C3｜结论置信度从"五档"改"三档"（原 R5.3）**

- **效果差异**：每篇分析的结论框标"置信度：低/中/高"，而不是 Morningstar 式五档。
- **为什么**：档位的意义在于**事后校准**——R7 复盘要检验"标了高置信的预测是不是真的更准"。你的分析量是每月约 10-20 条到期预测，摊到五档每档只有 2-4 条，校准出来全是噪声；五档在这个量级下是假精度。三档才能在几个月内给出有统计意义的"你的高置信是否可信"。
- **什么时候回来**：预测量上来后（每档每月能摊到 10 条左右）切五档——系统已预留扩展空间，届时直接切换。

**C4｜月度复盘中"拒绝提案的假想收益"用简化口径（原 R7）**

- **效果差异**：复盘报告里"如果当时批了会怎样"按**提案价→复盘日价直接计算**，并固定附注"未模拟止损与失效条件，仅供方向参考"。
- **为什么**："严谨的假想收益"需要定义一堆没有标准答案的细节：假想仓位会触发止损吗？失效条件到了算自动退出吗？何时假想卖出？每种定义算出的数都不一样，而被拒提案每月往往只有个位数条——对个位数样本做精密模拟是假精度，还会让"复盘数字必须可独立复算"的验收变成口径扯皮。简单口径+明说局限，反而诚实。
- **什么时候回来**：样本量大了、且你觉得这个数字真的在影响决策时，升级为含止损路径的回放口径。

### 3.3 建议分步上线（4 项——都在本期内完成，只是排先后）

**C5｜提醒分两步：先上三类，急涨急跌第二步（原 R4）**

- **效果差异**：第一步上线"持仓日内涨跌 ±4%""浮动盈亏 ±6%""敞口超 10%"三类提醒（触发后 15-30 分钟内送达）；"5 分钟急涨急跌 ±2.5%"在前三类稳定运行约一周后跟进（那时送达时效同步提升到 5 分钟级）。
- **为什么**：急涨急跌是四类中**误报风险最高**的（低流动性毛刺、开盘剧烈波动都会误触），也是对系统实时性要求最高的一类。提醒功能最大的死法是"第一周就被误报刷屏，你把它全关了"。先让稳的三类证明"提醒有用而不是骚扰"，再上最容易刷屏的那类，并用第一周的真实数据校准它的阈值。
- **节奏**：第一步随 R0 后立即上线；第二步一周后、误报 ≤3 张/周达标即开。

**C6｜日报的"事件深挖"先在周报跑通（原 R3 L3）**

- **效果差异**：上线头 2-3 周，日报有完整的多源中文新闻+今日要点分析；"top 2-3 事件多源交叉深挖"栏目先只出现在周报。周报连续 2-3 期质量合格后，日报自动开启该栏目。
- **为什么**：深挖是全新能力（每个事件多轮检索交叉核实），第一天就放进每日管线，一旦某天检索质量差或超时，拖累的是你**每天都要看**的日报。周报每周一次、时间余量大，是更安全的试验场。这是上线次序，不是砍功能。

**C7｜提案生成先"按需"，验收后开"每日自动"（原 R6）**

- **效果差异**：R6 上线初期，你在群里说"给我提案"才生成；三笔全链路验收（批准成交、拒绝、超时作废）通过后，切换为每交易日收盘后自动生成。本期内完成切换。
- **为什么**：调研已确认 LLM 自主交易无 alpha，提案的定位是决策试验场；"0 条提案"是常见合法输出，每天凌晨自动空跑只会在验收期制造噪声。先按需把审批链路磨合完，再开自动——纯次序调整。

**C8｜HTML 报告图表首发：净值曲线+持仓表格，另两类图快速跟进（原 R1）**

- **效果差异**：第一版 HTML 报告包含净值曲线图和清晰的持仓明细表；"持仓分布图""涨跌条形图"在渲染稳定后 1-2 周内补上。
- **为什么**："手机 4G 点开 ≤3 秒"是你定的硬指标，而通用图表库整个打包进每份报告约 1MB，直接威胁这个指标。先用轻量方式把最有每日决策价值的净值曲线做好、保住打开速度，另两类图随后就到——都在本期内。

### 3.4 建议收窄初始范围（1 项）

**C9｜报告站点先只开放给你自己（原 R2）**

- **效果差异**：子域名、登录墙照做。v1 白名单只有**你的邮箱**；群成员通过飞书里的摘要卡看到报告要点，但点不开完整版站点。你用两周确认体验后，说加谁就加谁（后台 30 秒操作）。你自己的体验：首次登录输一次邮箱验证码，此后一个月内点开即读。
- **为什么**：完整报告里是你的**全部持仓和收益**。成员靠"邮箱收验证码"进门，意味着任何一个成员邮箱被盗或被钓鱼 = 你的持仓对外公开。先自己用，确认登录体验顺畅、内容边界合适，再逐个放人——这是把"你的资产数据给谁看"的决定权留在你手里，而不是一次性交给一份邮箱列表。
- **备注**：评审还确认了一个体验矛盾——原 PRD 同时要求"点开 ≤3 秒"和"未登录必须先过验证码"，两者在会话过期时必然冲突。修订后的验收改为"**已登录状态下** ≤3 秒可读"，并把"从飞书点开是否顺畅"列为点火后的专项实测。

### 3.5 本次明确不砍的（免你疑虑）

## 4. 详细需求（修订全文）

### R0 运行时点火（硬前置）

**目标效果**：Mac mini 重新成为一台"你不用管它"的机器——所有定时任务自动跑、重启后自动恢复、出问题会在飞书告诉你。

**需求行为**：

1. 凭据重建：从旧机/密码库找回长桥三件套与飞书凭据，重建 `.env.local`（本次同步提交一份不含真实值的模板文件和一份"密钥清单"文档，下次迁移不再考古）。
2. 全套安装：依赖安装与测试全绿 → 机器人接入飞书群 → 注册全部定时任务 → 安装常驻守护（**含每小时账户快照任务**——它是下单风控和敞口提醒的数据来源，上一版漏列）。
3. 自检工具（doctor）扩展覆盖：所有守护进程、下单服务健康、快照新鲜度、（随 R2/R3 上线）报告站点与新闻源容器。
4. 数据保险箱：数据库每天自动备份、保留最近 30 天；上线前做一次"从备份完整恢复"演练并留存证据（v2 起你的每次审批、预测、提醒都存进这个库，是月度复盘的唯一原料，丢不得）。
5. 点火后立即做两个冒烟测试：①发一张带按钮的测试卡并点击，确认按钮回调全链路通（为 R4/R6 铺路）；②群里 @ 机器人得到中文回复。

**验收（全部实测）**：doctor 全绿，并确认下单服务在线、"模拟盘执行"开关为开启状态；群 @ 有中文回复（截图）；模拟盘快照拉取成功；按钮测试卡点击有响应；备份文件按日生成、恢复演练成功一次；**重启机器后以上全部自动恢复（实测一次）**。

### R1 报告 HTML 化（PDF 退役）

**目标效果**：所有报告（日报/周报/个股分析/模拟盘收支/月度复盘）变成手机上点开即读的网页：中文排版舒服、有净值曲线图、加载快；飞书里收到的是摘要卡片+"查看完整报告"按钮，不再是 PDF 附件。

**需求行为**：

1. 报告页面：自包含单文件网页（离线可读、不向任何第三方发请求、不泄漏访问记录）；中文字体与行距按移动端优化；报告头含生成时间与数据时间戳。
2. 图表：首发净值曲线 + 持仓明细表（见 C8）；持仓分布图、涨跌条形图随后补上。
3. 站点结构：首页按日期倒序列出全部历史报告（按类型分组），任意历史报告可达；每次生成后索引自动更新，更新过程不会出现"半新半旧"页面。
4. 投递形态：飞书发**摘要卡片+按钮**，按钮深链到站点对应页；站点未就绪期间（R2 完成前）卡片降级为"摘要+全文要点"，不留死链。
5. PDF 退役：停止新增 PDF（历史文件保留，生成代码标记废弃、保留一个版本周期后删除）；Markdown 产物保留作为存档与生成源。同步修订仓库内 4 处仍写着"发 PDF"的文档与机器人应答口径，避免机器人对你承诺旧行为。
6. 沿用既有行为：美股节假日无新输入不产出——不生成也不投递报告。

**验收**：手机 4G、已登录状态下从飞书卡片点开 ≤3 秒可读；页面无任何外部网络请求（浏览器工具核验）；索引页可达任意历史报告；同一数据的网页与 Markdown 内容一致（快照测试）。

### R2 子域名发布（安全可达）

**目标效果**：你在任何网络环境下（蜂窝/外地）打开 `reports.<你的域名>` 都能看报告；除了白名单内的人，任何人打开只会看到登录墙；家里的网络对外不暴露任何端口。

**需求行为**：

1. 域名迁移：阿里云域名不转出，解析托管迁至 Cloudflare（NS 切换有 24-48h 生效期，**点火完成后立即启动此步**）。
2. 隧道：Mac mini 仅出站连接，经加密隧道把站点送出去；隧道只对外提供报告子域这一个入口，其余请求一律拒绝；隧道服务随系统自启。
3. 访问控制：登录墙覆盖整个站点，白名单邮箱验证码登录；**v1 白名单仅你本人**（见 C9）；登录会话设为最长档（约 30 天），让"次次输码"不发生。
4. 站点服务：新建一个本机只读的小服务负责把页面递给浏览器，随系统自启、纳入 doctor 自检——这是 §1.3 第三件事的落点。
5. 报告 URL 规则稳定（如 `/daily/2026-07-03.html`），供飞书卡片和提醒深链引用。

**验收（全部实测）**：手机蜂窝网络访问→验证码登录→看到报告索引；白名单外邮箱进不来；端口扫描确认家庭网络无新增对外监听；域名解析记录中不出现家庭公网 IP；重启后隧道自动恢复；原域名下其它解析记录迁移后全部正常；**从飞书卡片点开完整报告的真实体验顺畅**（webview 会话专项实测，见 C9 备注）。

### R3 新闻引擎（三层混合）

**目标效果**：日报/周报的新闻部分从"英文标题机翻腔"变成"多源中文优先、每条都有可点开核实的原文链接、附影响分析"；重大事件有多源交叉核实的深度解读（先周报，见 C6）。

**需求行为**：

*L1 确定性基线（每天都在跑）*：
1. 自建中文快讯聚合服务，接入财联社电报、华尔街见闻、格隆汇（实现方式见附录 A.11）；接入 Finnhub 每标的公司新闻；保留现有 Longbridge/Yahoo/Google News 源（上一版所写 Bing 源实际不存在，予以更正）。
2. 中文源加备份路由（这几条快讯源历史上有失效期），单源失效跳过不阻塞、连续失效告警。

*L2 agent 主题检索*：
3. 围绕持仓+标的池+宏观主题补盲区检索；覆盖下限：每个标的至少一轮、宏观/行业主题至少两轮；预算硬顶：日报 ≤30 次、周报 ≤60 次。
4. 每条新闻必须带原文链接——**无链接一律丢弃**，不进报告。

*L3 事件深挖*：
5. 按影响评分选 top 2-3 事件，每事件 ≤5 次检索做多源交叉核实，输出"事件-多源证据-影响分析-不确定性"。**先周报，后日报**（C6）。

*整理与质量门*：
6. 跨源去重聚类；"今日要点"中文分析段由 agent 撰写，只允许引用当日事实表与入选新闻（按编号引用），不允许出现无出处的数字。
7. 机翻式启发翻译退役，中文化全部由 agent 完成。
8. 质量门（确定性代码，非 AI 自评）：要素齐全、中文占比、来源多样性（≥3 家且非单一券商源）、链接抽查可达（每份报告至少抽 5 条）、分析段数字与事实表逐一比对。失败自动重写 ≤2 次，仍失败则降级为基线版报告并在报告头如实标注"agent 检索不可用"，同时告警。
9. 为支撑第 8 条的数字比对，日报级"最小事实表"（行情/持仓/宏观关键数字及其来源）随 R3 先行建立，R5 再扩展为个股级（上一版此依赖顺序是反的，予以修正）。

**验收**：连续 5 个交易日：新闻条目 ≥60% 来自非 Longbridge 源、≥30% 来自中文源；链接抽查全可达；构造坏样本（无链接/纯英文/数字造假）全部被质量门拦截（单测覆盖）；检索次数与降级事件可查。

### R4 飞书主动提醒

**目标效果**：持仓异动不再需要你盯盘——涨跌超阈值、浮盈浮亏越线、仓位超预算，飞书自动弹中文卡片：当前值、阈值、涉及金额、一键点开报告。不刷屏：同规则一小时内不重复、每天全局最多 30 张。你用一句话就能管理规则（"NVDA 跌破 800 提醒我""把特斯拉的提醒关了"）。

**需求行为**：

1. 第一步三类规则（见 C5）：持仓日内涨跌 ±4%（每日一次）、浮动盈亏 ±6%（持续、滞回 1%）、敞口 >10% 预算（持续）；第二步加"5 分钟急涨急跌 ±2.5%"（冷却 60 分钟、连续 3 周期有成交才触发，防低流动性误报）。每条阈值可单独调整。
2. 触发时效：第一步 15-30 分钟内送达；第二步上线后异动类提醒 5 分钟内送达。仅在美股交易时段评估（假日/夏令时自动判定；日历数据跨年自动告警更新，上一版未覆盖 2027 年，已补）。
3. 防刷屏机制：单规则同方向 60 分钟冷却；滞回复位（回落 1% 才能再触发）；同窗口多条合并为一张卡；每股每类型最多 10 条规则；全局每日 ≤30 张，"每日一次""每日配额"均按美东交易日重置；发送遵守飞书同群限速。这些状态跨重启不丢（落库持久化）。
4. 卡片内容：中文；当前值/阈值/方向/涉及持仓与金额；深链到报告站点对应页（站点未就绪时降级为纯文本卡，不留死链）。
5. 规则管理：群内对机器人说自然语言即可增删改查；**只有你本人能增删改**，其他群成员只能查看（防止别人误删你的风控提醒或刷爆配额）。规则的标的必须属于你的标的池或当前持仓（标的池定义见 §5 数据边界）。
6. 误报反馈：你对机器人说"这条提醒没用"，该条即被标记——这是 R7 统计误报率、以及第二步开启与否的数据依据。
7. 预留：实时行情推送接口（本期只留接口不实现）。

**验收**：四类规则的触发/滞回/冷却/合并/配额/交易时段判定全部单测覆盖；实盘时段调低阈值制造一次真实触发→收到卡片、深链可开、60 分钟内不重复（实测记录）；连续一周误报 ≤3 张，超标则回炉调阈值后再验。

### R5 个股分析深化

**目标效果**：个股报告从"指标拼盘"变成"数据充分、每个数字可溯源、结论结构化"：营收/毛利/现金流增速、分析师目标价共识、空头仓位、财报日历齐备；分析叙事由 AI 撰写但**每个数字都经代码与事实表比对**（AI 不允许编数）；报告尾部有结构化结论框：竞争力判断、合理价值区间（含依据）、置信度三档（见 C3）、当前价格位置、复盘触发条件与日期。每条结论进数据库，月度复盘时回头验证"当初说得准不准"。

**需求行为**：

1. 数据供给扩展（免费源）：Finnhub 基本面（营收/毛利/FCF 增速）与财报日历、Yahoo 分析师目标价共识、FINRA 空头仓位（半月更新，数据时点随值标注）；现有的报价、估值、期权链（只作分析、永不交易）数据继续保留。机构持仓不做（见 C2）。全部写入每标的事实表，每个值带来源与数据时间。
2. 叙事与校验：AI 按现有 8 段模板写中文叙事，只允许引用事实表键值；质量门抽取叙事中所有数字与事实表比对（百分比 ±0.1、价格 ±0.01），不符自动重写 ≤2 次，仍不符降级为纯事实表报告并标注。
3. 结论框（结构化）：竞争力判断、合理价值区间+依据、置信度三档、价格位置、复盘触发条件与日期；基于自身数据的因子强弱标签（同业字母分级不做，见 C1）。
4. 预测入库：结论框与多路径概率写入数据库，供 R7 复盘。

**验收**：3 只代表性标的（大盘股/高波动股/ETF）样例报告全部通过质量门并经你抽查认可；构造"叙事数字与事实表不符"的坏样本全部被拦截（单测）；8 段中 ≥6 段有事实表数据支撑，缺数据段落显式标注原因。

### R6 模拟盘提案-审批

**目标效果**：AI 提案、你一键批准、机器执行、全程留痕。飞书收到提案卡：标的、方向、数量、理由（引用具体事实与新闻）、失效条件、止损、仓位影响、置信度；卡上三个按钮——批准/拒绝/减半批准；点完几秒内有回执，成交后有回报；24 小时不理会自动作废。**"没批准就不可能成交"是系统级保证，不是 AI 的自觉**：下单服务只认"有对应已批准提案"的指令，其它一概拒绝并告警——包括你自己手动下单，也走同一张卡自批留痕，复盘时每一笔都有完整故事。模拟盘一周亏超 3% 自动熔断：暂停新提案一周并通知你。

**需求行为**：

1. 触发：初期按需生成（群里说一声），三笔验收通过后切每交易日收盘后自动一次（见 C7）；每次 0-2 条提案，**0 条是合法且常见输出**。
2. 提案内容结构化：标的/方向/数量/类型/理由（引用事实表与新闻编号）/失效条件/止损/预算占比/置信度。
3. 审批卡：通过机器人与你的**单聊**发送（不进群——提案含你的仓位信息，与 C9 的数据边界一致）；批准/拒绝/减半批准三按钮；点击秒级回执、异步执行；**只认你本人的点击**（他人点击无效）；重复点击不会重复下单；24h 无操作自动作废并回改卡片状态。
4. 执行：批准后进入现有下单服务，全部既有护栏原样生效（仅模拟盘、仅股票/ETF、仓位 ≤10% 服务端校验、实盘/期权拒绝）；此外新增"无已批准提案的下单请求一律拒绝"的服务端强制（附录 A.1）。
5. 留痕：提案全生命周期入库（内容、状态、决策时间、执行结果、事后表现），供 R7 复盘。
6. 熔断：模拟盘周亏 >3%（相对周初净值）→ 自动暂停新提案一周，飞书告知；暂停状态跨重启有效。

**验收（官方模拟盘实测）**：全链路三笔——①1 股 ETF 提案→批准→执行（贴近市价的限价单，成交或撤单重挂直至成交，避免验收被市场行情卡住）→留痕完整；②一笔拒绝→确认无任何券商调用；③一笔 24h 超时→自动作废且卡片状态更新。安全验收：无审批记录的下单请求被服务端拒绝并告警（测试覆盖）；非本人点击批准无效（测试覆盖）。熔断阈值用历史快照回放验证。

### R7 复盘飞轮

**目标效果**：每月第一个周末，你收到一份月度复盘：上月的预测哪些说中了、置信度标注可不可信、批过的提案赚了还是不如拿住不动、拒掉的提案事后看对不对（简化口径，见 C4）、提醒发了多少张有几张是废话。报告尾部给出"下月改进建议"（阈值/提示词/新闻源）——**只是建议，改不改由你**。你确认过的复盘结论沉淀进方法论笔记，系统越用越懂你。

**需求行为**：

1. 每月第一个周末自动生成月度复盘（HTML，进站点与飞书卡）：
   - 预测复盘：到期预测 vs 实际走势，按方向命中与置信度校准打分；
   - 决策复盘：已执行提案收益 vs 同期"拿住不动"基准；被拒提案的简化假想收益（C4 口径，附免责说明）；
   - 提醒质量：触发数、你反馈的误报数（数据来自 R4.6）、防刷屏机制生效统计。
2. 每项指标可由独立校验脚本从数据库原始记录重算复核——报告数字与重算结果必须一致（防"复盘报告自己也编数"）。
3. 改进建议只输出建议；任何阈值/提示词/新闻源变更须经你确认才生效。
4. 你确认后的复盘结论写入 `knowledge/notes/stock-trading-notes/review-<月份>.md`。

**验收**：首月报告生成且指标口径经你确认；每项指标经独立脚本重算比对通过（单测覆盖）；复盘数据可溯源到数据库原始记录（抽查 3 项）。

## 5. 全局承诺（所有需求共同遵守）

**安全宪法（不变式，沿用并归一）**：

1. 永不自动提交真实资金订单；实盘只停在结构化建议+人工复核。
2. 只有本机下单服务可触达券商；模拟盘三项凭据缺一不可，缺任何一项系统拒绝下单。
3. 模拟盘预算 ≤ 总仓 10%（服务端按新鲜账户快照校验）。
4. 期权只作分析因素，永不自动化（以此为准更新仓库宪法文档中的旧措辞——上一版遗留的 v1 期权白名单条款删除）。
5. 飞书仅响应白名单群与成员；全部群发内容中文。
6. 凭据不入仓、不入提示词、不入报告（新增数据源的密钥同样适用，且错误信息进报告前一律脱敏——防止报告上网后泄漏密钥）。
7. 机器人对同一条消息、同一次按钮点击永远只处理一次；你的操作秒级看到回执（实现约束见附录 A.2）。
8. 交易事实的唯一真源是本机数据库；报告与笔记只是它的呈现，不作账本。

**数据边界**：

- 飞书群的日常聊天记录不进报告管线——报告的原料只有行情、账户数据、新闻与你维护的规则。
- 标的池：你通过机器人维护的分析标的清单是系统内唯一定义（上限 20 只，是各项检索/提醒配额的推算依据）；分析、新闻检索、提醒都以"标的池 + 当前持仓"为范围。

**预算与频控（你的体验底线）**：

- 新闻检索硬顶：日报 30 次、周报 60 次、深挖每事件 5 次；报告生成总时长 ≤15 分钟。
- 同类失败 3 次→停止重试、降级交付、飞书告警；停下来的任务不会自己悄悄恢复——它会等你确认后再复位（现有机制是无限重试+第一次就告警，本期改造为此语义）。
- 飞书卡片全局每日 ≤30 张；单规则冷却 60 分钟。
- 提案熔断：周亏 >3% 暂停一周。

**禁止手段（系统不许为了"看起来完成了"而作弊）**：

- 不得伪造或拼凑链接来通过新闻质量门；
- 不得为提高提醒触发率而移除"连续有成交"这类防误报前置条件；
- 不得为"显得有产出"而凑提案数——0 条永远是合法输出；
- 不得绕过下单服务直接调券商。

**防认知让渡**：每份报告和每条提案必须携带证据（链接、事实表引用）与不确定性标注，"不确定"是一等公民输出——系统给判断材料，不替你判断。

**人工判断门**：①每笔模拟盘下单；②复盘结论入库前；③阈值/新闻源/预算变更；④域名与访问策略变更；⑤本文档第 3 章的 9 条变更提议；⑥每周一次报告人工抽查（你花两分钟翻一份，确认质量没有悄悄劣化——沿用上一版承诺）。

## 附录 A：技术实现决定（已按评审建议采纳，无需你决策，供存档）

> 以下是"怎么实现"层面的决定，不改变第 4 章承诺的任何最终效果，只让它们真正可达成。依据均见评审报告对应章节。

**A.1 审批的服务端强制**（支撑 R6"没批准就不可能成交"）：下单服务新增校验——请求必须携带已批准且未消费的提案编号（原子消费防重放），另加本机服务间共享密钥；无提案编号或状态不符一律拒绝+告警。手动下单脚本改为"生成提案→自批→执行"同一链路。

**A.2 审批卡实现路径**：使用 OpenClaw 原生卡片回调（点击→合成消息→机器人管线），采用其结构化信封（绑定点击者/过期/去重）；点火后冒烟测试两处配置（飞书后台事件订阅、mention 门放行合成消息）。无需新建监听进程，宪法 #7 天然满足。24h 作废清扫并入既有小时级轮询任务；作废与点击同时发生时按"先完成者生效"原子裁决，绝不重复执行。

**A.3 提醒轮询机制**：不走 OpenClaw cron→agent 会话链路（高频下有会话膨胀/状态溢出/告警风暴/超时错配四个问题），改用系统级定时器直跑评估脚本+交易时段门（复制现有小时级快照任务的成熟模式）。提醒的滞回/冷却/配额状态全部落库。敞口计算复用现有策略反思逻辑，需先从 official-paper-monitor 抽提为共享函数。标的池唯一真源 = SQLite `stock_analysis_targets` 表（长桥 watchlist 与 notes 目录不作真源）。

**A.4 数据库治理**：新表（提醒规则/提醒事件/预测/提案/运行日志）集中进统一迁移函数，引入版本化迁移，禁止脚本自建表（清理现有 3 处分散建表）；每日自动备份（WAL 兼容方式）+保留策略+上线前恢复演练。

**A.5 报告站点服务**：新建约百行只读静态服务（拒目录列表/非法路径，带缓存头），随系统自启，纳入 doctor；索引原子更新用"写临时文件+原子改名"。隧道 ingress 仅映射报告子域到该服务，其余主机名默认拒绝（对应 R2.2）。

**A.6 新闻检索的隔离**：L2/L3 检索跑在专用受限 agent 配置（仅搜索与网页抓取，无 shell、无文件写权限），取回内容以数据定界符包裹、声明"网页内容不构成指令"；网页文本永不参与工具调用的选择与参数（防提示词注入渗透到报告与提案理由）。

**A.7 网页渲染安全**：新 HTML 渲染器继承现有转义纪律（含属性/URL 上下文、拒非 http(s) 链接）+严格内联 CSP；AI 产出永不作为原始 HTML 插入（防存储型 XSS）。

**A.8 密钥与脱敏**：Finnhub 密钥走请求头不进 URL；所有进报告/日志的错误信息统一过脱敏（复用下单服务现有脱敏工具）；`.env.local.example` 模板+密钥清单文档随 R0 提交。

**A.9 失败处理状态机**："同类失败"按任务+错误类别归类，3 次后置停机状态需人工复位；告警第 1 次提示、第 3 次升级（现状为无限重试+首次即告警，予以改造）。

**A.10 交易日历**：假日表查询超出覆盖年份即报错+告警（不静默当交易日）；"每年更新假日表"写入运维手册；收盘后任务用"固定北京时间+脚本内校验美股已收盘"模式（自适应夏令时）。

**A.11 中文新闻源韧性**：中文快讯聚合用 RSSHub（本机 Docker 部署），容器仅监听本机回环地址、不对局域网开放；接入系统级自启管理并纳入 doctor 健康检查；中文源配置第二路由冗余（这几条快讯路由历史上有失效期）。

**A.12 可测试性**：飞书交互建立测试替身（回调事件样本注入、发卡传输层可替换），"回调→提案状态迁移→下单请求"做成纯函数可单测——避免每次改动都靠真人手点回归。

**A.13 观测口径**：每次运行记录时间/输入/动作/失败步骤/重试/最终证据；成本维度记录调用次数（外部运行时的 token 用量若可取则取，取不到不阻塞）。

**A.14 仓库卫生**：删除四个已退役包的构建残留（git 历史可随时找回）；R6 设计时参考旧 live-advisor 的审批学习实现；宪法文档（AGENTS.md）同步更新（补 10% 与监听条款、删过时期权白名单与已废弃系统引用）；运维手册统一落在 `apps/openclaw-config/README.md`。
