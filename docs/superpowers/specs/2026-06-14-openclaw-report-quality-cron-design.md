# OpenClaw Report Quality Cron Design

## Goal

Make OpenClaw, not direct launchd report scripts, own scheduled trading report production. Every daily, weekly, and stock-analysis output must be generated by a repeatable pipeline whose quality checks force the generator to meet the user's requirements instead of treating bad output as acceptable.

## Requirements

- Register report jobs in OpenClaw cron so `openclaw cron list`, `openclaw cron run`, and `openclaw cron runs` are the operational source of truth.
- Keep launchd only for durable OpenClaw/Gateway style services where needed, not as the main report scheduler.
- Daily, weekly, and stock-analysis reports must share a consistent Chinese PDF typography stack.
- News coverage must use diversified sources. Longbridge may be one source, but reports must also attempt outside sources and preserve source evidence.
- News lines must be detailed enough to audit: Chinese title/summary, media or publisher, channel, original title when relevant, URL or source index, and impact.
- English source material must be summarized or translated into Chinese; long English-only content is not acceptable.
- Stock analysis must combine price action, PE/PB or explicit missing-data reasons, target/upside evidence, trend, and option-chain pressure. Option-chain evidence is important but not sufficient by itself.
- Quality checks are regression gates for the generator. A failing check means the code path that produced the report must be fixed and re-run until future OpenClaw outputs pass.
- Runtime validation may still prevent accidental delivery of malformed output, but the expected end state is valid generation, not a permanent “failed quality” notification workflow.
- Feishu group mention handling must stay allowlisted and keep the trading safety constitution intact.

## Approach

The implementation will add focused tests first for report quality and OpenClaw cron registration. The report generators will then be changed until those tests pass. The runtime report commands will run validation immediately after generation, before delivery, so manual and OpenClaw-triggered runs exercise the same quality rules.

OpenClaw cron installation will be a repository command that removes or updates managed cron jobs and registers the retained jobs with clear names. Cron jobs write schedule-marker system events into OpenClaw's run ledger. A local loopback runner service watches `~/.openclaw/cron/runs/*.jsonl`, deduplicates run IDs, and executes the same repository scripts used by tests and manual smoke checks. This keeps OpenClaw cron as the schedule and run source of truth while avoiding direct launchd report schedules and avoiding OpenClaw webhook delivery that is blocked by the gateway SSRF guard for loopback URLs.

## Acceptance

- `pnpm test` passes.
- A generated daily report passes the validator and no longer accepts Longbridge-only news when outside sources were unavailable without a structured remediation path in the generator.
- A generated weekly report passes the same source/detail checks.
- A stock-analysis report includes valuation and upside evidence, not only option-chain commentary.
- `openclaw cron list` shows managed jobs for daily, weekly, and stock analysis.
- `openclaw cron run <job>` can be used as the verification entry point for OpenClaw-owned production; the runner must observe the new run-log entry and execute the corresponding quality pipeline.
