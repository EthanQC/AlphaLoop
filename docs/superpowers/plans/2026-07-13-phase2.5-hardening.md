### Task 3: Migration v7——数据完整性（表重建）

**问题**：三处结构性缺口，P6 双账户一落地就会出错：
1. `stock_analysis_targets.symbol` 是主键 → **结构上无法表达 per-owner 标的池**（spec 要求每人各自维护、每人上限 20）。
2. `alert_events.owner_id` 无 FK；`members.status` 无 CHECK；`alert_rules.direction` 无 CHECK（拼错值会被引擎静默当 'both'）。
3. `official_paper_snapshots` 等表的 owner_id 可空且无索引约束——历史行 NULL 是合法的（共享账户），但新写入必须带 owner。

**Files:**
- Modify: `packages/shared-types/src/database.ts`（MIGRATIONS[6] = v7；SCHEMA_VERSION → 7）
- Test: `packages/shared-types/src/database.test.ts`

**Interfaces（v7 迁移内容，全部在一个事务内）:**
1. **stock_analysis_targets 重建**为 per-owner：新表 `(symbol TEXT NOT NULL, owner_id TEXT REFERENCES members(id), active INTEGER NOT NULL DEFAULT 1, created_at TEXT NOT NULL, updated_at TEXT NOT NULL, PRIMARY KEY (symbol, owner_id))` — 注意 SQLite 的复合主键允许 owner_id 为 NULL 且 NULL 值不参与唯一性判断，会导致同 symbol 多行 NULL。**因此 owner_id 必须 NOT NULL**：迁移时把现有 NULL 行的 owner_id 回填为**一个哨兵值 `'__legacy_shared__'`**（不引用 members，故 FK 需允许——改为不加 FK，只加 CHECK(owner_id <> '')），并在代码层把该哨兵解释为"共享池，任何成员可见"。实现者若认为有更干净的方案（如回填为第一个 active member 的 id，无成员时保留哨兵），可提出并在报告中论证，但**必须保证迁移后 (symbol, owner_id) 唯一且无数据丢失**。
2. **alert_rules 重建**：加 `CHECK(direction IN ('both','up','down'))`，其余列与约束原样保留（含 v6 的 removed_at）。
3. **members 重建**：加 `CHECK(status IN ('active','revoked'))`。
4. **alert_events 重建**：`owner_id TEXT NOT NULL REFERENCES members(id)`。
5. 表重建标准流程：`PRAGMA foreign_keys=OFF`（迁移期间）→ CREATE new_x → INSERT INTO new_x SELECT ... FROM x → DROP x → ALTER TABLE new_x RENAME TO x → 重建索引 → `PRAGMA foreign_keys=ON`。**注意**：migrate() 的每步已包在事务里；`PRAGMA foreign_keys` 在事务内无效——需在 migrate() 框架层面处理（实现者：读 migrate() 实现，若必须在事务外设 pragma，就在 migrate() 里为需要它的步骤提供支持，并保证失败仍能回滚；把方案写进报告）。

- [ ] Step 1: 失败测试（迁移后各表约束生效：坏 direction/坏 status/无 owner 的 event 均被拒；legacy 数据全保；(symbol,owner_id) 唯一；旧库 v6→v7 无损）
- [ ] Step 2-5: TDD → Commit `feat: schema v7 - per-owner watchlist, check constraints and event ownership`
